# 第一部分：产品需求文档（PRD）

---

## 1. 产品概述

RimLog（边缘日志）是一个以"非人类视角观察人类日常"为核心的内容产品。用户打开网页或小程序，看到的是一位驻扎在银河系猎户臂末端的外星观察员（编号 ORN-7741）发回的"未授权考察报告"。观察员以极其认真、略带困惑的科研口吻，记录人类的日常行为——上班、开会、刷手机、焦虑、喝奶茶——但由于完全不理解人类文化语境，得出的结论荒诞而温柔。

这不是一个教人做事的产品，不是心理疏导工具，不是知识付费。它是一面镜子，让人从宇宙的尺度重新看自己，看完之后觉得自己的焦虑挺可爱的，然后今晚好过一点。

## 2. 目标用户

**核心用户画像：** 25-35 岁互联网从业者（尤其是程序员、产品经理、设计师），对 AI 发展有清醒认知，对未来职业路径感到迷茫和焦虑，但不需要被"指导"或"激励"。他们每晚躺在床上刷手机消磨焦虑，需要一种不那么有毒的内容替代品。

**扩展用户：** 任何在时代转型期感到"被卡住"的年轻人。不限行业。焦虑是普遍的，这个产品的切入点是普适的。

## 3. 核心价值主张

- 不贩卖焦虑，不提供解决方案，不教人做事
- 通过视角转换（非人类观察者）实现情绪释放——把恐惧变成幽默，把焦虑变成可爱
- 让用户在宇宙尺度下重新审视自己的日常，获得一种"原来我的烦恼在银河系看来挺好笑的"的轻松感
- 核心情绪价值：不是"别怕"，是"你看，人类这个物种挺有意思的，包括正在焦虑的你"

## 4. 产品形态

### 4.1 主产品：Web 网页（rimlog.earth）

一个沉浸式的深空风格网页，模拟一个银河边缘观测站的终端界面。

**核心交互——多时间尺度日志系统：**

| 时间尺度 | 内容类型 | 更新频率 | 示例 |
|---------|---------|---------|------|
| 1 天 | 逐小时的实时观察，贴合真实时间 | 每日更新 | "08:15 大规模定向迁移开始，个体面部表情：空洞" |
| 100 天 | 近期社会现象观察 | 每周 1-2 篇 | "观察到一种叫年终奖的资源分配仪式" |
| 1 年 | 季节性/周期性行为记录 | 每月 1 篇 | "每年第一天人类集体许愿，第七天集体放弃" |
| 10 年 | 趋势分析与技术变迁 | 每季度 1 篇 | "通讯能力越强，孤独感越强" |
| 100 年 | 文明级别的观察 | 不定期 | "硬件升级了，操作系统没变" |
| 1000 年 | 物种级别的终极困惑 | 不定期 | "观察1000年后仍无法理解人类为什么把水烧热泡叶子" |

**页面元素：**

- 深空星场背景（Canvas 动态渲染，星星闪烁）
- 星云渐变叠加层，营造深空氛围
- CRT 扫描线效果，模拟老旧观测站终端
- 实时地表时间同步显示
- 观察员状态栏（信号状态、累计观察周期、观测站位置）
- 时间尺度切换器（顶部 tab bar）
- 日志条目流（带时间戳、分类标签、正文、观察员私人备注）
- 每条日志附带"观察员备注"，用更私人的语气补充——这是人格化的关键，让观察员不只是一个记录器，而是一个有困惑、有情绪、有幽默感的"人"
- 底部 footer：观测站坐标、观察员信息（实习期第 4612 恒星周期，仍未转正）

**视觉风格关键词：** 深空蓝黑、科幻终端、冷色辉光、等宽字体、低饱和度、克制而精致

### 4.2 内容分发渠道

| 渠道 | 形式 | 用途 |
|------|------|------|
| 小红书 | 图文卡片（截取单条日志 + 深空背景） | 拉新引流，制造传播 |
| 微信公众号 | 每日推送 1 条日志 | 留存，建立阅读习惯 |
| 即刻 | 短文 + 链接 | 程序员社区冷启动 |
| Twitter/X | 英文版日志 | 国际化探索 |

小红书和公众号的内容是网页内容的子集——每天取 1 条最有传播力的日志，做成适合对应平台的格式。所有渠道引流回主站。

### 4.3 未来可能的产品拓展（不急，先验证核心）

- **用户投稿翻译器：** 用户提交自己的日常（"今天加班到11点改了一个按钮颜色"），系统以观察员口吻重新"翻译"（"一个碳基生命消耗14小时的生命来改变若干光子的波长"）
- **观察员语音日志：** 音频版，带太空通讯的噪音和失真效果，适合睡前听
- **实体周边：** "未授权观察报告"实体笔记本、贴纸（"请勿上报总部"）
- **多观察员视角：** 引入不同编号的观察员，对同一事件给出不同解读，丰富世界观

## 5. 内容生产策略

### 5.1 内容调性定义（极其重要）

**是什么：**
- 严肃的科研口吻——像真的在写论文/田野报告
- 真诚的困惑——观察员是真的不懂，不是装不懂
- 幽默来自"太认真"，不来自"抖机灵"
- 温柔的底色——观察员对人类有一种说不清的喜欢，虽然它不理解人类

**不是什么：**
- 不是段子手/搞笑博主
- 不是毒鸡汤
- 不是科普（不解释科学知识）
- 不是说教（绝不暗示"你应该怎么做"）
- 不是居高临下（观察员自己也是个焦虑的实习生）

### 5.2 角色设定

**观察员 ORN-7741：**
- 银河联邦（如果存在的话）派驻猎户臂末端的田野观察员
- 实习生身份，已实习 4612 个恒星周期，仍未转正
- 观察地球是它的工作任务，但这些报告是"未授权"的——它私下写的，不敢发给总部，因为总部对人类的态度是"建议消灭"，而它觉得人类挺有意思的
- 性格：极其认真、略带社恐、对人类行为充满困惑但又忍不住继续观察
- 它自己也有职场焦虑（实习4600年还没转正），但它不会直接说，只偶尔在备注里流露

### 5.3 生产流程

1. **素材收集：** 从真实生活、社交媒体热点、程序员社区中提取人类行为素材
2. **视角转换：** 将行为用观察员口吻重新描述——去掉所有人类预设知识，假装完全不懂
3. **AI 辅助写作：** 用 LLM 生成初稿，但必须人工精调。调性控制是这个产品的命脉，不能完全交给 AI
4. **备注撰写：** 每条日志附带的"观察员备注"是灵魂——这里放最个人化、最幽默、最温柔的内容
5. **质量标准：** 每条日志必须同时满足——(a) 让人笑，(b) 让人有一瞬间从自己的视角里跳出来，(c) 不说教

### 5.4 内容日历（第一个月）

- 第 1 周：每日 1 条"1天尺度"日志，跑通调性，收集反馈
- 第 2 周：引入"100天尺度"，一周 2 篇，开始小红书分发
- 第 3 周：引入"1年尺度"，同时开公众号
- 第 4 周：全尺度内容上线，评估数据，决定后续方向

## 7. 风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 调性跑偏，变成普通段子号 | 失去独特性，用户流失 | 建立严格的内容审核标准，宁可少发不乱发 |
| 用户觉得内容重复 | 新鲜感消退 | 多时间尺度保证内容纵深；引入时事观察保持时效性 |
| 被大号搬运/抄袭 | 原创价值被稀释 | 加强品牌IP建设，观察员 ORN-7741 是不可复制的角色资产 |
| AI 生成的内容太"像AI" | 调性不对，失去灵魂 | 人工精调是必须的，AI只辅助不主导 |

## 2. 技术选型

### 2.1 前端

| 模块 | 选型 | 理由 |

## 3. 数据模型

### 3.1 日志条目（log_entries）

```sql
CREATE TABLE log_entries (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  
  -- 内容
  title         TEXT,                          -- 可选标题
  content       TEXT NOT NULL,                 -- 正文（观察员口吻）
  observer_note TEXT,                          -- 观察员私人备注
  
  -- 分类
  timescale     VARCHAR(10) NOT NULL,          -- '1d', '100d', '1y', '10y', '100y', '1000y'
  tag           VARCHAR(20) NOT NULL,          -- 'routine', 'anomaly', 'insight', 'critical'
  tag_label     VARCHAR(20) NOT NULL,          -- 中文标签显示文本
  
  -- 时间
  display_time  VARCHAR(50) NOT NULL,          -- 显示用时间文本，如 "08:15 地表时间"
  publish_at    TIMESTAMPTZ NOT NULL,          -- 实际发布时间
  
  -- 元数据
  is_published  BOOLEAN DEFAULT FALSE,
  sort_order    INTEGER DEFAULT 0,            -- 同尺度内排序
  
  created_at    TIMESTAMPTZ DEFAULT NOW(),
  updated_at    TIMESTAMPTZ DEFAULT NOW()
);

-- 索引
CREATE INDEX idx_timescale_published ON log_entries(timescale, is_published, publish_at DESC);
```

### 3.2 用户投稿（Phase 2）

```sql
CREATE TABLE user_submissions (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  
  original_text TEXT NOT NULL,                 -- 用户原文
  translated    TEXT,                          -- 观察员翻译后的文本
  status        VARCHAR(20) DEFAULT 'pending', -- 'pending', 'translated', 'published', 'rejected'
  
  created_at    TIMESTAMPTZ DEFAULT NOW()
);
```

## 6. 1天尺度的"实时感"实现

核心体验：用户在早上8点打开，看到观察员记录的8点事件；晚上11点打开，看到晚间日志。仿佛观察员真的在实时观察。

实现方式：

- 每条"1天尺度"的日志在创建时设定一个 `display_time`（如 08:15）
- 后端 API `/api/logs/today` 只返回 `display_time <= 当前时间` 的条目
- 前端轮询或 SSE 推送，每隔一段时间检查是否有新条目
- 视觉上可加入"新信号接收中..."的过渡动画

注意：不是真的实时生成，是预写好当天全部日志（6-8条），按时间点逐步释放。对用户来说效果等同于实时。
